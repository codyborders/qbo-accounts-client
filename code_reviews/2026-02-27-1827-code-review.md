## Summary

This review covers 7 bug fixes applied to the `qbo-accounts-client` library via strict TDD. The fixes address: exception handling crashes, OAuth2 resource leaks, rate-limit retry ordering, incorrect inheritance, missing model fields, and HTTP-date header parsing.

**Key takeaways:**
- All fixes are correct, minimal, and well-tested.
- Three medium-severity issues were found: (1) `close()` not declared on the `AuthHandler` ABC, (2) redundant type guard in exception parsing, and (3) inline stdlib import that should be at module level.
- The code-simplifier agent already applied fixes for items 2 and 3, plus cleaned up test imports.
- Security and performance agents surfaced longer-term architectural recommendations (retry loop, thread safety, header case-insensitivity) that are out of scope for this bug bash but worth tracking.

## Design Findings

*(From Phase 1: Software Design Principles review)*

### DESIGN-1: `close()` Not Declared on `AuthHandler` ABC (Medium)

**What changed:**
A `close()` method was added to `OAuth2Auth` (`auth.py:83-85`), and `QBOClient.close()` was updated to call it (`client.py:220-224`). However, the call is gated behind an `isinstance` check:

```python
# client.py lines 220-224
def close(self) -> None:
    """Close the underlying HTTP client and auth resources."""
    self._client.close()
    if isinstance(self.auth, OAuth2Auth):    # <-- couples to concrete class
        self.auth.close()
```

**How it works:**
When `QBOClient.close()` is called, it first closes its own HTTP connection pool (`self._client`), then checks if the auth handler is specifically an `OAuth2Auth` instance. Only if it is does it call `auth.close()`. If someone writes a custom `AuthHandler` subclass that also holds resources (e.g., a custom token provider with its own HTTP client), `QBOClient.close()` would silently skip its cleanup.

**Why this matters:**
This violates the Open/Closed Principle -- adding a new auth handler type that needs cleanup requires modifying `QBOClient.close()`. The fix is to add a default no-op `close()` on the `AuthHandler` ABC, then remove the `isinstance` guard:

```python
# auth.py -- add to AuthHandler
class AuthHandler(ABC):
    @abstractmethod
    def apply(self, request: httpx.Request) -> httpx.Request: ...

    def close(self) -> None:
        """Release resources held by the auth handler. No-op by default."""

# client.py -- simplify close()
def close(self) -> None:
    self._client.close()
    self.auth.close()    # Always safe -- base class has no-op default
```

### DESIGN-2: Test Couples to Private Attribute (Minor)

**What changed:**
`tests/test_client.py:211` asserts on `auth._http_client.is_closed` to verify cleanup:

```python
def test_close_closes_oauth2_http_client(self):
    ...
    client.close()
    assert auth._http_client.is_closed    # <-- private attribute access
```

**Why this matters:**
If `OAuth2Auth` renames or restructures its internal HTTP client, this test breaks even though the behavior (closing resources) hasn't changed. A better approach would be to add an `is_closed` property to `OAuth2Auth` or mock `_http_client.close()` and assert it was called.

### DESIGN-3: Duplicated Case-Insensitive Header Lookup (Minor)

**What changed:**
`utils.py:16` and `utils.py:31` both use the pattern `headers.get("Header-Name") or headers.get("header-name")`:

```python
retry_after = headers.get("Retry-After") or headers.get("retry-after")
...
reset = headers.get("X-RateLimit-Reset") or headers.get("x-ratelimit-reset")
```

**Why this matters:**
HTTP headers are case-insensitive per RFC 7230. The `httpx.Headers` object already handles this, but `client.py:201` converts to a plain `dict` with `dict(response.headers)`, which loses case-insensitivity. The manual two-variant lookup is a workaround for this unnecessary conversion. The root fix is to pass `response.headers` directly (which is already a `Mapping[str, str]`) and update the type hint to `Mapping[str, str]`.

## Security Findings

*(From Security Sentinel agent)*

### SEC-1: QBO Query Injection Filter Uses Blocklist (Medium)

**What exists (not changed in this PR):**
`resources/base.py:19-26` validates query parameters with a regex blocklist (`; -- /* */`). This can be bypassed with single quotes, `UNION`, or newline injection.

**How it works:**
The `_validate_query_param()` function uses a regex to reject dangerous patterns. It checks for semicolons, double dashes, and block comment markers. However, strings like `Name = 'test' OR '1'='1'` pass the filter because single quotes and SQL keywords are not blocked.

**Why this matters:**
While QBO's own API likely rejects malformed queries server-side, defense-in-depth means the client should not send potentially dangerous strings. An allowlist approach (only alphanumeric, spaces, comparison operators, parentheses) would be more secure. This is pre-existing and not part of this PR's changes, so it is informational.

### SEC-2: OAuth2 Tokens Stored as Public Attributes (Low)

**What exists:**
`auth.py:46-47` stores `access_token` and `refresh_token` as public attributes, while `_client_id` and `_client_secret` were correctly privatized with `_` prefix.

**Why this matters:**
If the auth object is accidentally logged or serialized (e.g., via `repr()` or debug output), tokens would be exposed. Adding `__repr__` redaction and privatizing the token attributes would be consistent with the existing pattern for `_client_id`/`_client_secret`.

### SEC-3: Token Refresh Race Condition Under Threading (Low)

**What exists:**
`client.py:205-207` calls `self.auth.refresh()` without locking. If multiple threads share one `QBOClient` and both receive 401 simultaneously, both will attempt to refresh. The second thread's refresh will use a stale refresh token (Intuit rotates them on use), causing failure.

**Why this matters:**
This is a synchronous client, so concurrent use is unlikely but possible. Either add a `threading.Lock` or document that `QBOClient` is not thread-safe.

### SEC-4: `realm_id` Not Validated (Low)

**What exists:**
`client.py:157` interpolates `realm_id` into URL paths without validation. QBO company IDs are numeric strings. Adding `realm_id.isdigit()` validation in `__init__` would prevent path traversal attempts.

## Performance Findings

*(From Performance Oracle agent)*

### PERF-1: Single-Retry on 429 Is Fragile (Medium)

**What changed:**
`client.py:199-207` retries once on 429, then falls through. If the retry also returns 429 (common during burst throttling), the caller gets `RateLimitError`.

**How it works:**
The code sends a request. If it gets 429, it sleeps for `Retry-After` seconds and retries once. If the retry also returns 429, it is not retried -- it falls through to the error handler. Additionally, if the initial request returns 401 and the refresh retry returns 429, that 429 is not handled either.

**Why this matters:**
At high request volumes (batch imports, bulk queries), this causes intermittent `RateLimitError` exceptions. A bounded retry loop with configurable max retries would be more robust. This is an architectural improvement beyond the scope of the current bug bash.

### PERF-2: `dict(response.headers)` Copies Headers Unnecessarily (Low)

**What changed:**
`client.py:201` creates a plain `dict` copy of response headers before passing to `RateLimiter`:

```python
self._rate_limiter.wait_if_needed(dict(response.headers))
```

**How it works:**
`response.headers` is an `httpx.Headers` object (a `MutableMapping[str, str]`). The `wait_if_needed` method only calls `.get()` on it, so the copy is unnecessary. Worse, the copy loses `httpx.Headers`' built-in case-insensitive lookup.

**Why this matters:**
Minor performance impact (allocates a dict of ~20-40 entries per 429 response), but the real issue is losing case-insensitivity, which forces the manual two-variant `.get()` fallback in `utils.py`.

### PERF-3: Lazy Import in Hot Path (Low)

**What changed:**
`utils.py:22` imports `parsedate_to_datetime` inside a nested `except` block:

```python
except (ValueError, TypeError):
    try:
        from email.utils import parsedate_to_datetime    # <-- lazy import
```

**How it works:**
Python's `import` statement acquires a global import lock (`importlib._bootstrap._find_and_load`) on the first call. After the first import, it is cached and fast. However, the first invocation in a threaded application could cause lock contention.

**Why this matters:**
`email.utils` is a stdlib module with negligible import cost (~0.3ms). Moving it to the module level removes one level of nesting and eliminates the theoretical lock contention. The code-simplifier agent already applied this fix.

## Code Quality Findings

*(From Kieran Python Reviewer agent)*

### QUAL-1: `auth.refresh()` Failure Leaks Raw Exception (Medium)

**What changed:**
`client.py:206` calls `self.auth.refresh()` which internally does `response.raise_for_status()` (auth.py:74). If the token endpoint returns an error (e.g., 400 invalid_grant, network timeout), the caller gets a raw `httpx.HTTPStatusError` instead of a clean `AuthenticationError`.

**How it works:**
The `refresh()` method sends a POST to Intuit's token endpoint, calls `raise_for_status()` on the response, and parses the JSON. If the refresh fails, the exception propagates up through `QBOClient.request()` with no wrapping. The caller sees an `httpx` exception that reveals nothing about the QBO context.

**Why this matters:**
Callers expect `QBOClient.request()` to raise `QBOError` subclasses. An unwrapped `httpx.HTTPStatusError` breaks that contract. Wrapping with `AuthenticationError("Token refresh failed", ...) from exc` would be more consistent.

### QUAL-2: `conftest.py` Client Fixture Does Not Clean Up (Low)

**What exists (not changed in this PR):**
`tests/conftest.py` creates a `QBOClient` in the `client` fixture but does not call `close()` afterward. Since `httpx.Client` holds connection pools, this leaks file descriptors across the test suite.

**How to fix:**
```python
@pytest.fixture
def client(auth: BearerAuth) -> Iterator[QBOClient]:
    c = QBOClient(realm_id=REALM_ID, auth=auth, base_url=BASE_URL)
    yield c
    c.close()
```

## Simplification Findings

*(From Code Simplifier agent -- changes already applied)*

### SIMP-1: Redundant `isinstance(errors, list)` Guard Removed

**File:** `qbo_accounts/exceptions.py:78`

**Before:**
```python
errors = fault.get("Error", [])
if isinstance(errors, dict):
    errors = [errors]
if errors and isinstance(errors, list):    # <-- redundant check
    first = errors[0]
```

**After:**
```python
errors = fault.get("Error", [])
if isinstance(errors, dict):
    errors = [errors]
if errors:    # <-- simplified: errors is guaranteed to be a list here
    first = errors[0]
```

**Why:** After `fault.get("Error", [])`, `errors` is either a `dict` (normalized to a list on the next line), a `list` (the expected case), or `[]` (the default). After line 77, it is always a list. The `isinstance(errors, list)` check was redundant.

### SIMP-2: `parsedate_to_datetime` Import Moved to Module Level

**File:** `qbo_accounts/utils.py`

**Before:**
```python
except (ValueError, TypeError):
    try:
        from email.utils import parsedate_to_datetime    # <-- nested import
        retry_dt = parsedate_to_datetime(retry_after)
```

**After:**
```python
import time
from email.utils import parsedate_to_datetime    # <-- top-level import

...
except (ValueError, TypeError):
    try:
        retry_dt = parsedate_to_datetime(retry_after)
```

**Why:** `email.utils` is a standard library module with negligible import cost. Inline imports are appropriate for heavy optional dependencies, not stdlib modules. Moving it to the top reduces nesting and eliminates import lock contention in threaded scenarios.

### SIMP-3: Test Imports Lifted to Module Level

**File:** `tests/test_client.py` -- `map_status_to_exception` import moved from inside `test_fault_error_as_dict()` to the module-level import block alongside other exception imports.

**File:** `tests/test_models.py` -- `PurchaseOrderCreate` and `DepartmentUpdate` imports moved from inside test methods to the module-level import block alongside `QBOBaseModel`.

**Why:** All other imports in these files are at the module level. Local imports inside test methods are inconsistent with the established pattern. Module-level imports also ensure import errors are caught at collection time, not at test execution time.

## Items Not Addressed (Out of Scope)

The following were flagged by reviewers but are architectural improvements beyond the scope of this bug bash. They should be tracked separately:

1. **Retry loop refactor** (PERF-1): Replace sequential if-checks with a bounded retry loop handling 429/401 in any order.
2. **`AuthHandler.close()` ABC method** (DESIGN-1): Add default no-op `close()` to base class.
3. **Thread safety** (SEC-3): Add `threading.Lock` to refresh flow or document non-thread-safe usage.
4. **Query injection allowlist** (SEC-1): Replace blocklist validator with allowlist approach.
5. **Token attribute privatization** (SEC-2): Rename `access_token`/`refresh_token` to `_access_token`/`_refresh_token`.
6. **Pass `response.headers` directly** (PERF-2/DESIGN-3): Avoid `dict()` conversion to preserve case-insensitivity.
7. **Client fixture cleanup** (QUAL-2): Convert `conftest.py` client fixture to yield fixture with `close()`.
8. **Wrap refresh failures** (QUAL-1): Catch `httpx` errors in `auth.refresh()` and re-raise as `AuthenticationError`.
