# Code Review Report -- 2026-03-13

## Summary

- **Repository**: codyborders/qbo-accounts-client
- **Date**: 2026-03-13 15:30
- **Files reviewed**: 12 (7 production, 5 test)
- **Review agents**: Software Design, Code Simplifier, Kieran Python Reviewer, Security Sentinel, Performance Oracle

This review covers the bug bash commit that fixed 7 bugs across the QBO Accounts client library. The codebase is well-structured with a clean generic resource pattern and good test coverage (205 tests). The most actionable findings are: (1) incomplete query injection sanitization, (2) duplicated SQL-building logic between `BaseResource` and `_SystemResourceBase`, (3) `_parse_json` not validating the parsed value is a dict, and (4) single-retry rate limit handling that will fail at scale.

## Design Findings

### D1: Duplicated SQL-building logic (Medium)

**Files**: `qbo_accounts/resources/base.py` (lines 101-114) and `qbo_accounts/resources/system.py` (lines 30-50)

`_SystemResourceBase._execute_query()` reconstructs the same SQL-building logic that `BaseResource._build_query()` already provides. Both methods build `SELECT * FROM {entity}` with optional WHERE/ORDER BY clauses and run `_validate_query_param` on each. If the query construction pattern changes (e.g., new injection protection, new clauses), both must be updated in sync.

Notably, `_execute_query` in system.py does **not** strip pagination clauses like `BaseResource.query()` now does -- so the Bug 3 fix is incomplete for system resources.

**How the code works:**

In `base.py`, `_build_query()` constructs SQL:
```python
def _build_query(self, where=None, order_by=None) -> str:
    sql = f"SELECT * FROM {self.QUERY_ENTITY}"
    if where:
        _validate_query_param(where, "where")
        sql += f" WHERE {where}"
    if order_by:
        _validate_query_param(order_by, "order_by")
        sql += f" ORDER BY {order_by}"
    return sql
```

In `system.py`, `_execute_query()` duplicates this same logic inline:
```python
sql = f"SELECT * FROM {query_entity}"
if where:
    _validate_query_param(where, "where")
    sql += f" WHERE {where}"
if order_by:
    _validate_query_param(order_by, "order_by")
    sql += f" ORDER BY {order_by}"
sql += f" STARTPOSITION {start_position} MAXRESULTS {max_results}"
```

**Why it matters:** This is an information leakage red flag -- the same design decision (how to build a QBO SQL query) lives in two modules. A standalone `build_query()` function in `base.py` could serve both resource hierarchies.

### D2: Nonobvious `inspect.signature` in CLI read command (Low)

**File**: `qbo_accounts/cli.py`, lines 140-147

The `read` command introspects `resource.read` via `inspect.signature()` to detect whether the method requires an entity ID argument. This handles both singleton resources (CompanyInfo, Preferences -- no ID) and entity resources (Invoice -- requires ID).

```python
sig = inspect.signature(resource.read)
requires_id = any(
    p.default is inspect.Parameter.empty
    and p.name != "self"
    and p.kind in (inspect.Parameter.POSITIONAL_ONLY, inspect.Parameter.POSITIONAL_OR_KEYWORD)
    for p in sig.parameters.values()
)
```

**Why it matters:** A reader must understand Python's `inspect` module to follow this logic. A simpler approach would be a class-level `REQUIRES_ID = True` flag on resources, checked with `getattr(resource, 'REQUIRES_ID', True)`. The current approach is defensible but adds cognitive overhead.

### D3: Inconsistent payload wrapping between resource hierarchies (Low)

**Files**: `qbo_accounts/resources/base.py` (line 97) vs `qbo_accounts/resources/system.py` (line 83)

`BaseResource.update()` sends the serialized body directly:
```python
body = data.model_dump(by_alias=True, exclude_none=True)
resp = self._client.request("POST", path, json=body)
```

`CompanyInfoResource.update()` wraps it in an extra key:
```python
payload = {self.ENTITY_KEY: data.model_dump(by_alias=True, exclude_none=True)}
resp = self._client.request("POST", path, json=payload)
```

This reflects a genuine QBO API difference, but the inconsistency is undocumented. A comment explaining why system resources wrap payloads differently would prevent confusion.

## Security Findings

### S1: Incomplete query injection sanitization (Medium)

**File**: `qbo_accounts/resources/base.py`, lines 19-26

The `_DANGEROUS_PATTERN` regex blocks `;`, `--`, `/*`, and `*/`, but uses a denylist approach that is inherently incomplete:

```python
_DANGEROUS_PATTERN = re.compile(r"(;|--|[/][*]|[*][/])")
```

**What it misses:**
- Single-quote breakout: `DisplayName = 'x' OR 1='1'` passes validation
- UNION-style queries: `UNION SELECT * FROM Account` contains no blocked patterns
- Keyword injection: `Active = true ORDER BY Id` in the `where` parameter appends an extra ORDER BY

**Mitigating factor:** QBO's server-side query parser is not a full SQL engine -- it rejects most true SQL injection attacks. The risk is primarily query manipulation (data exfiltration across entity boundaries), not arbitrary server-side execution.

**Recommendation:** Add keyword blocking for SQL manipulation keywords (`UNION`, `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `DROP`) when they appear outside of expected contexts, or implement a positive-validation allowlist of field names and operators.

### S2: Credentials stored as mutable public attributes (Medium)

**File**: `qbo_accounts/auth.py`, lines 47-48

Both `access_token` and `refresh_token` are stored as plain public attributes on the `OAuth2Auth` instance:

```python
self.access_token = access_token
self.refresh_token = refresh_token
```

Tokens would appear in `repr()` output, debugging dumps, or exception tracebacks. If the object is accidentally serialized (e.g., pickled for logging), tokens are exposed.

**Recommendation:** Override `__repr__` and `__str__` to redact sensitive fields. Consider using Pydantic `SecretStr` or a similar wrapper.

### S3: `extra="allow"` on input models passes unvalidated data to API (Low)

**File**: `qbo_accounts/models/base.py`, line 13

All models inherit `extra="allow"`, meaning arbitrary key-value pairs in input JSON are silently accepted and passed through to the QBO API. While intentional for forward-compatibility with new API fields, it means CLI user-supplied JSON bypasses all Pydantic type validation for any key not explicitly defined in the model.

**Recommendation:** Consider using `extra="forbid"` on create/update models (user input boundary) while keeping `extra="allow"` on read models (API response boundary).

### S4: No input validation on entity_id path parameter (Low)

**File**: `qbo_accounts/resources/base.py`, line 90

The `entity_id` is incorporated directly into the URL path without validation:

```python
path = self._client._build_path(self.ENTITY, entity_id)
```

While `httpx` handles URL encoding and QBO's server would reject malformed paths, a defense-in-depth approach would validate that `entity_id` contains only alphanumeric characters.

### S5: Token persistence race condition (Low)

**File**: `qbo_accounts/cli.py`, lines 33-38

Two `set_key` calls write to `.env` non-atomically. If the process is killed between the two writes, the access token is updated but the refresh token is stale. The old refresh token may have been invalidated, locking the user out.

```python
set_key(dotenv_path, "QBO_ACCESS_TOKEN", access_token)   # written
set_key(dotenv_path, "QBO_REFRESH_TOKEN", refresh_token)  # process killed here
```

### S6: Unbounded `max_results` parameter (Low)

**File**: `qbo_accounts/resources/base.py`, lines 116-128

The `max_results` parameter accepts any integer, including very large values (999999999) or zero/negative values. Should be clamped to a valid range (1-1000).

## Performance Findings

### P1: Single-retry rate limit handling will fail at scale (High)

**File**: `qbo_accounts/client.py`, lines 199-202

The `request()` method retries exactly once on a 429 status. If the second attempt also returns 429, the request fails. For `query_all` pagination loops through thousands of entities, hitting the QBO rate limit will cause hard failures mid-stream.

| Scenario | Items | Estimated Time | Bottleneck |
|---|---|---|---|
| Small company | 100-500 | < 5s | Network latency |
| Medium company | 1,000-5,000 | 10-50s | Pydantic validation |
| Large company | 10,000+ | 2-10 min | Rate limiting -- single retry will fail |

**Recommendation:** Implement 3-5 retries with exponential backoff and jitter, respecting `Retry-After` headers.

### P2: Redundant pagination regex in `query()` (Low)

**File**: `qbo_accounts/resources/base.py`, line 125

```python
sql = _PAGINATION_CLAUSE_RE.sub("", sql).strip()
```

In `query()`, the SQL was just built by `_build_query()` which never inserts pagination clauses, so this regex substitution is always a no-op. The regex is rightfully needed in `auto_paginate_query()` (pagination.py) to clean user-supplied queries, but in `query()` it is wasted work.

**Why it was added:** Bug 3 addressed the scenario where a user includes `STARTPOSITION` in a `where` clause. However, `_build_query` constructs the WHERE clause from the `where` parameter, so the pagination text would only appear if the user explicitly passed it in `where`. The regex strip is defensive but adds unnecessary overhead on the hot path.

### P3: Per-item `model_validate` in `query_all` is the CPU bottleneck (Low)

**File**: `qbo_accounts/resources/base.py`, lines 143-144

Every item from `auto_paginate_query` is individually validated through `model_validate`. For 10,000 items, that is ~1-2 seconds of pure CPU time. When the caller is the CLI (which ultimately serializes back to JSON via `_serialize` -> `model_dump`), the round-trip of dict -> Pydantic model -> dict is pure overhead.

**Recommendation:** Consider offering a `query_all_raw()` method that yields raw dicts for bulk export scenarios.

### P4: `_serialize` recursion redundant after `model_dump` (Low)

**File**: `qbo_accounts/cli.py`, lines 94-102

The `_serialize()` function recursively walks every object tree. But `model_dump(by_alias=True)` already returns plain dicts with no nested Pydantic models, so the recursive dict/list traversal checking for `BaseModel` instances is always a no-op for model output.

## Code Quality Findings

### Q1: `_parse_json` does not validate parsed value is a dict (Bug)

**File**: `qbo_accounts/cli.py`, lines 80-85

```python
def _parse_json(raw: str) -> dict[str, Any]:
    try:
        return json.loads(raw)
    except json.JSONDecodeError as e:
        _error(f"Invalid JSON: {e}")
```

`json.loads` can return any JSON value -- a list, string, number, not just a dict. If someone passes a valid JSON array like `'[1,2,3]'` to `create`, it will fail with a confusing Pydantic validation error rather than a clear CLI error.

**Fix:**
```python
def _parse_json(raw: str) -> dict[str, Any]:
    try:
        result = json.loads(raw)
    except json.JSONDecodeError as e:
        _error(f"Invalid JSON: {e}")
    if not isinstance(result, dict):
        _error(f"Expected a JSON object, got {type(result).__name__}")
    return result
```

### Q2: Bare `dict` type annotations throughout models (Medium)

**Files**: `namelist.py`, `system.py`, `transactions.py` (dozens of locations)

Fields like `primary_email_addr: dict | None`, `line: list[dict]`, and `accounting_info_prefs: dict | None` appear throughout. Bare `dict` is the type-hint equivalent of `Any`. At minimum these should be `dict[str, Any]` to be explicit about key types.

The most common ones (especially `Line` items, which are central to transaction behavior) deserve their own Pydantic models or at least a `TypeAlias`.

### Q3: Mixed `hasattr`/`isinstance` capability checking in CLI (Medium)

**File**: `qbo_accounts/cli.py`

Three commands (`create`, `read`, `query`) use `_require_capability` with `hasattr`:
```python
_require_capability(resource, entity, "create")  # line 188
```

Three commands (`delete`, `deactivate`, `void`) use direct `isinstance` checks:
```python
if not isinstance(resource, TransactionResource):  # line 223
```

This inconsistency is confusing. Since well-defined resource classes exist with known methods, all commands should use the same pattern.

### Q4: `_output_stream` produces non-standard JSON formatting (Low)

**File**: `qbo_accounts/cli.py`, lines 110-117

The streaming output writes commas on separate lines:
```
[
{"Id": "1"}
,
{"Id": "2"}
]
```

This is valid JSON but non-standard formatting. Consider emitting the comma as a prefix on subsequent items.

### Q5: `Class_` naming with trailing underscore (Low)

**File**: `qbo_accounts/models/namelist.py`, line 12

`Class_` avoids the Python `class` keyword collision. This is acceptable per PEP 8, but `QBOClass` or `ClassEntity` would be more descriptive and conventional.

### Q6: No `__all__` exports in model modules (Low)

**Files**: `namelist.py`, `system.py`, `transactions.py`

These modules export many classes. Without `__all__`, it is unclear which classes are part of the public API.

## Simplification Findings

### SM1: `create` and `update` CLI commands share identical structure (Medium)

**File**: `qbo_accounts/cli.py`, lines 183-212

Both commands follow the same pattern: parse JSON, check `isinstance(resource, BaseResource)`, validate into model, call method. A shared helper would eliminate 6-8 lines of duplication:

```python
def _mutate(resource, data, method_name, cls_attr):
    if isinstance(resource, BaseResource):
        model = getattr(resource, cls_attr).model_validate(data)
        return getattr(resource, method_name)(model)
    return getattr(resource, method_name)(data)
```

### SM2: `delete`/`deactivate`/`void` commands share identical structure (Low)

**File**: `qbo_accounts/cli.py`, lines 215-254

All three follow: get resource, check isinstance, call method with `(entity_id, sync_token)`, output result.

### SM3: `_get_resource` patch repeated 13 times in tests (Low)

**File**: `tests/test_cli.py`

Nearly every test method wraps its core logic in `with patch("qbo_accounts.cli._get_resource", return_value=mock_resource)`. A shared pytest fixture would reduce boilerplate.

### SM4: Repeated mock setup in rate limiter tests (Low)

**File**: `tests/test_utils.py`

Every test method sets `mock_time.time.return_value = 1740700000.0` and `mock_time.sleep = MagicMock()`. This setup could be extracted into a shared fixture.
