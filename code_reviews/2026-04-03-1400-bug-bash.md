### Bug 1: OAuth state param never validated (CSRF)

**Detail Bug ID**: `bug_86d20cb2-d39f-4475-905f-8c9eb3508851`
**File**: `qbo_accounts/oauth.py`
**Security vulnerability**: Yes
**Impact**: Medium | **Severity**: Critical | **Priority**: P0
**Status**: Fixed

#### What was the problem?

The OAuth2 authorization flow generated a random `state` parameter (used for CSRF protection per RFC 6749), but never actually validated it. The `build_auth_url()` function created the state value and embedded it in the URL, but only returned the URL string -- callers had no way to store the expected state. The callback server in `run_callback_server()` accepted any authorization code without checking whether the returned `state` matched.

This means an attacker could trick a user into authorizing the attacker's QuickBooks account by sending a crafted callback URL (e.g., `http://localhost:8484/callback?code=ATTACKER_CODE`) to the user's browser while the callback server was listening.

#### What was changed?

**Before:**
```python
def build_auth_url(client_id: str) -> str:
    params = {
        "client_id": client_id,
        "redirect_uri": _REDIRECT_URI,
        "response_type": "code",
        "scope": "com.intuit.quickbooks.accounting",
        "state": secrets.token_urlsafe(16),
    }
    return f"{_INTUIT_AUTH_URL}?{urlencode(params)}"

def run_callback_server() -> str:
    result: dict[str, str] = {}
    class Handler(BaseHTTPRequestHandler):
        def do_GET(self) -> None:
            qs = parse_qs(urlparse(self.path).query)
            if "code" in qs:
                result["code"] = qs["code"][0]  # accepts any code
```

**After:**
```python
def build_auth_url(client_id: str) -> tuple[str, str]:
    state = secrets.token_urlsafe(16)
    params = {
        "client_id": client_id,
        "redirect_uri": _REDIRECT_URI,
        "response_type": "code",
        "scope": "com.intuit.quickbooks.accounting",
        "state": state,
    }
    return f"{_INTUIT_AUTH_URL}?{urlencode(params)}", state

def run_callback_server(expected_state: str) -> str:
    result: dict[str, str] = {}
    class Handler(BaseHTTPRequestHandler):
        def do_GET(self) -> None:
            qs = parse_qs(urlparse(self.path).query)
            returned_state = qs.get("state", [None])[0]
            if returned_state != expected_state:
                result["error"] = "state_mismatch"
                # ... rejects the request
            elif "code" in qs:
                result["code"] = qs["code"][0]
```

The CLI caller in `cli.py` was updated to destructure the tuple:
```python
url, state = build_auth_url(client_id)
code = run_callback_server(state)
```

#### How does the code work?

1. `build_auth_url()` now returns a tuple of `(url, state)` instead of just the URL string. This gives the caller access to the generated state value.
2. `run_callback_server()` now requires an `expected_state` parameter. Before processing the authorization code, it extracts the `state` from the callback query string and compares it against the expected value.
3. If the state does not match, the handler responds with HTTP 403 and sets `result["error"]` to `"state_mismatch"`, which causes the function to raise a `RuntimeError` just like any other OAuth error.
4. Only if the state matches does the handler proceed to extract the authorization code.

#### Why was this change needed?

The OAuth 2.0 specification (RFC 6749 Section 10.12) requires clients to implement CSRF protection using the `state` parameter. Without this validation, an attacker could perform an OAuth CSRF attack where they authorize access to their own resources and trick the victim into using the attacker's tokens. This fix closes that security gap by completing the state validation loop.

### Bug 2: Auth refresh raises HTTPStatusError not QBOError

**Detail Bug ID**: `bug_429aa15b-a165-46d9-8153-fb8a42f49bde`
**File**: `qbo_accounts/auth.py`
**Security vulnerability**: No
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

When `QBOClient.request()` received a 401 response and triggered automatic token refresh via `OAuth2Auth.refresh()`, and that refresh failed (e.g., because the refresh token was revoked), the library raised a raw `httpx.HTTPStatusError` instead of an `AuthenticationError` from the library's own exception hierarchy (`QBOError`). This meant code that caught `QBOError` to handle all API errors would miss refresh failures, causing unexpected crashes.

#### What was changed?

**Before:**
```python
response = self._http_client.post(self._token_url, ...)
response.raise_for_status()  # raises httpx.HTTPStatusError on 4xx/5xx
```

**After:**
```python
response = self._http_client.post(self._token_url, ...)
try:
    response.raise_for_status()
except httpx.HTTPStatusError as exc:
    raise AuthenticationError(
        message=f"Token refresh failed: HTTP {response.status_code}",
        status_code=response.status_code,
    ) from exc
```

#### How does the code work?

1. The `refresh()` method still calls `response.raise_for_status()` to detect HTTP errors.
2. If the status check fails, the `except` block catches the `httpx.HTTPStatusError`.
3. It wraps the error in an `AuthenticationError` (a subclass of `QBOError`), preserving the original exception as the cause via `from exc`.
4. Callers catching `QBOError` or `AuthenticationError` now correctly handle refresh failures.

#### Why was this change needed?

The library establishes a contract that all API-related errors are subclasses of `QBOError`. When `refresh()` let `httpx.HTTPStatusError` escape, it violated this contract. Any code doing `try: client.request(...) except QBOError:` would fail to catch refresh errors, leading to unhandled crashes during normal token expiry scenarios.

### Bug 3: Pagination regex corrupts string literals

**Detail Bug ID**: `bug_a46d35df-7c75-4d84-a30c-6d978040f68d`
**File**: `qbo_accounts/pagination.py`, `qbo_accounts/resources/base.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The pagination clause stripping used a simple regex (`\s+(STARTPOSITION|MAXRESULTS)\s+\d+`) applied to the entire query string, including text inside single-quoted string literals. If a user queried for an entity with a name containing "STARTPOSITION" or "MAXRESULTS" followed by digits, the regex would silently remove that text from inside the quoted value, corrupting the search term.

For example: `WHERE Name = 'Test STARTPOSITION 5'` would become `WHERE Name = 'Test'`, returning wrong results with no error.

#### What was changed?

**Before:**
```python
# pagination.py
cleaned = _PAGINATION_CLAUSE_RE.sub("", base_query).strip()

# resources/base.py
sql = _PAGINATION_CLAUSE_RE.sub("", sql).strip()
```

**After:**
```python
# pagination.py -- new function
def strip_pagination_clauses(sql: str) -> str:
    segments = _QUOTED_SEGMENT_RE.split(sql)
    result_parts: list[str] = []
    for segment in segments:
        if segment.startswith("'"):
            result_parts.append(segment)  # preserve quoted text
        else:
            result_parts.append(_PAGINATION_CLAUSE_RE.sub("", segment))
    return "".join(result_parts).strip()

# Both call sites now use:
cleaned = strip_pagination_clauses(base_query)
```

#### How does the code work?

1. A new regex `_QUOTED_SEGMENT_RE` splits the SQL string on single-quoted boundaries, handling escaped quotes (`''`).
2. The `strip_pagination_clauses()` function iterates over the segments. Segments starting with `'` are inside quotes and are preserved verbatim.
3. Segments outside quotes have the pagination regex applied normally.
4. The segments are joined back together, and the result is stripped of leading/trailing whitespace.
5. Both call sites (`pagination.py` and `resources/base.py`) now use this function instead of the raw regex.

#### Why was this change needed?

Silent data corruption is one of the worst categories of bugs -- the user gets wrong results with no indication anything went wrong. While the edge case (a literal string containing pagination keywords) is uncommon, it could affect any user whose data happens to contain those terms. The fix preserves the original stripping behavior for real pagination clauses while protecting quoted string content.

### Bug 4: No 429 retry after OAuth token refresh

**Detail Bug ID**: `bug_2716f095-c080-4a95-b972-892a226a4d09`
**File**: `qbo_accounts/client.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The `QBOClient.request()` method had a retry loop for 429 (rate limit) responses on the initial request path, but when a 401 triggered a token refresh and the subsequent request received a 429, that response was not retried. Instead, it fell through to the error-raising code, causing a `RateLimitError` that should have been recoverable.

#### What was changed?

**Before:**
```python
response = self._send_authenticated(method, path, params, json, headers)

# Handle 429 with retry loop
max_retries = 3
for _ in range(max_retries):
    if response.status_code != 429:
        break
    self._rate_limiter.wait_if_needed(dict(response.headers))
    response = self._send_authenticated(method, path, params, json, headers)

if response.status_code == 401 and isinstance(self.auth, OAuth2Auth):
    self.auth.refresh()
    response = self._send_authenticated(method, path, params, json, headers)
    # no 429 retry here
```

**After:**
```python
def _retry_on_rate_limit(self, response, method, path, params, json, headers, max_retries=3):
    for _ in range(max_retries):
        if response.status_code != 429:
            break
        self._rate_limiter.wait_if_needed(dict(response.headers))
        response = self._send_authenticated(method, path, params, json, headers)
    return response

# In request():
response = self._send_authenticated(method, path, params, json, headers)
response = self._retry_on_rate_limit(response, method, path, params, json, headers)

if response.status_code == 401 and isinstance(self.auth, OAuth2Auth):
    self.auth.refresh()
    response = self._send_authenticated(method, path, params, json, headers)
    response = self._retry_on_rate_limit(response, method, path, params, json, headers)
```

#### How does the code work?

1. The 429 retry logic was extracted into a `_retry_on_rate_limit()` helper method.
2. This method is called after both the initial request and the post-token-refresh request.
3. Both code paths now have identical retry behavior: up to 3 retries on 429 responses, with rate limiter wait times respected.

#### Why was this change needed?

Rate limiting is common in production, especially right after a token refresh when the new token might be rate-limited by Intuit's servers. Without this fix, a perfectly recoverable 429 after refresh would crash the caller. The fix ensures consistent retry behavior regardless of the request path.

### Bug 5: CLI read crashes with TypeError on singletons

**Detail Bug ID**: `bug_a0dc69e1-9503-4aa2-96cf-08d89edd6800`
**File**: `qbo_accounts/cli.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

When a user ran `qbo read preferences 123` (passing an ID to a singleton resource like Preferences, Entitlements, or CompanyInfo), the CLI crashed with a Python `TypeError` traceback instead of producing a JSON error message. The signature inspection logic only validated the "missing ID" case but not the "extra ID" case.

#### What was changed?

**Before:**
```python
sig = inspect.signature(resource.read)
requires_id = any(
    p.default is inspect.Parameter.empty
    and p.name != "self"
    and p.kind in (inspect.Parameter.POSITIONAL_ONLY, inspect.Parameter.POSITIONAL_OR_KEYWORD)
    for p in sig.parameters.values()
)
if requires_id and entity_id is None:
    _error(f"'{entity}' requires an ID argument for read")
```

**After:**
```python
sig = inspect.signature(resource.read)
positional_params = [
    p for p in sig.parameters.values()
    if p.name != "self"
    and p.kind in (inspect.Parameter.POSITIONAL_ONLY, inspect.Parameter.POSITIONAL_OR_KEYWORD)
]
requires_id = any(p.default is inspect.Parameter.empty for p in positional_params)
if requires_id and entity_id is None:
    _error(f"'{entity}' requires an ID argument for read")
if not positional_params and entity_id is not None:
    _error(f"'{entity}' does not accept an ID argument")
```

#### How does the code work?

1. The positional parameters are collected into a list instead of being checked inline.
2. The existing "requires ID but none provided" check remains.
3. A new symmetric check is added: if the `read()` method has no positional parameters at all (i.e., it is a singleton resource), but the user provided an entity ID, the CLI emits a JSON error: `"'preferences' does not accept an ID argument"`.

#### Why was this change needed?

The CLI's contract (documented in the README) states that errors are written as JSON to stderr with a non-zero exit code. A raw Python traceback breaks scripts that parse JSON errors from stderr and provides a confusing experience for CLI users.

### Bug 6: Query parsing crashes when entity field is null

**Detail Bug ID**: `bug_6f9bbb06-9ac2-40b6-89a6-d5f023a5162f`
**File**: `qbo_accounts/models/base.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Major | **Priority**: P2
**Status**: Fixed

#### What was the problem?

`GenericQueryResponse.from_qbo_response()` crashed with `TypeError: object of type 'NoneType' has no len()` when the QBO API returned a response where the entity key existed but had a `None` value (e.g., `{"QueryResponse": {"Account": null}}`). The `dict.get(key, default)` method only uses the default when the key is missing entirely, not when the key exists with a `None` value.

#### What was changed?

**Before:**
```python
items_data = qr.get(entity_key, [])  # returns None if key exists with null value
```

**After:**
```python
items_data = qr.get(entity_key) or []  # handles both missing key and null value
```

#### How does the code work?

1. `qr.get(entity_key)` returns `None` if the key is missing or if the value is explicitly `None`.
2. The `or []` expression converts any falsy value (including `None`) to an empty list.
3. This matches the behavior of `len(items_data)` on the next line, which now always receives a list.

#### Why was this change needed?

QBO API responses can sometimes include entity keys with null values, especially for edge cases like empty result sets or certain query conditions. The library should handle these gracefully instead of crashing. Other parts of the codebase (like `pagination.py` and `exceptions.py`) already handle similar null/unexpected-type scenarios defensively.

### Bug 7: System resources duplicate pagination clauses

**Detail Bug ID**: `bug_28f98178-1cb9-47ed-9919-0893c942da6b`
**File**: `qbo_accounts/resources/system.py`
**Security vulnerability**: No
**Impact**: Low | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

`_SystemResourceBase._execute_query()` (used by BudgetsResource and ExchangeRatesResource) appended `STARTPOSITION` and `MAXRESULTS` clauses without first stripping any existing pagination clauses from user-supplied `where` or `order_by` parameters. If a user's input contained pagination keywords, the generated query would have duplicate clauses.

#### What was changed?

**Before:**
```python
sql += f" STARTPOSITION {start_position} MAXRESULTS {max_results}"
```

**After:**
```python
sql = strip_pagination_clauses(sql)
sql += f" STARTPOSITION {start_position} MAXRESULTS {max_results}"
```

#### How does the code work?

1. The `strip_pagination_clauses()` function (introduced in Bug 3 fix) is imported from `pagination.py`.
2. Before appending the controlled pagination clauses, any existing pagination keywords in the assembled SQL are removed.
3. This matches the behavior already present in `BaseResource.query()`.

#### Why was this change needed?

Consistency across all resource types. `BaseResource.query()` already stripped pagination clauses, but `_SystemResourceBase._execute_query()` did not. This created a behavioral discrepancy where Budgets and ExchangeRates queries could produce malformed SQL with duplicate pagination, while all other entity queries would not.

### Bug 8: Negative Retry-After triggers 5s default delay

**Detail Bug ID**: `bug_3c0d9a0b-240d-4aee-a122-ff1214dce580`
**File**: `qbo_accounts/utils.py`
**Security vulnerability**: No
**Impact**: Low | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

When the `Retry-After` header contained a negative numeric value, `time.sleep()` raised `ValueError`. The `except (ValueError, TypeError)` handler caught this silently and fell through to the HTTP date parsing path (which also failed), ultimately falling back to `DEFAULT_RETRY_DELAY` (5 seconds). A negative value is semantically equivalent to "retry immediately" (0 seconds), so the 5-second delay was unnecessary.

#### What was changed?

**Before:**
```python
time.sleep(min(float(retry_after), MAX_RETRY_AFTER))
```

**After:**
```python
wait = max(0.0, min(float(retry_after), MAX_RETRY_AFTER))
time.sleep(wait)
```

#### How does the code work?

1. `float(retry_after)` converts the header value to a number.
2. `min(..., MAX_RETRY_AFTER)` caps the wait at the maximum allowed value (300 seconds).
3. `max(0.0, ...)` clamps negative values to zero, preventing `ValueError` from `time.sleep()`.
4. This matches the defensive clamping already used in the HTTP date and X-RateLimit-Reset paths.

#### Why was this change needed?

Consistency with the other two header-handling paths, which both use `max(0.0, ...)` to clamp negative waits. Without this fix, a negative `Retry-After` header (which could come from a misconfigured server) imposes an unnecessary 5-second penalty on every retried request.

### Bug 9: VendorUpdate missing company_name field

**Detail Bug ID**: `bug_cc60fe8c-b197-4ad1-a264-ed912006f7c0`
**File**: `qbo_accounts/models/namelist.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

The `VendorUpdate` Pydantic model was missing the `company_name` field. This field exists on `Vendor` (read model), `VendorCreate`, and `CustomerUpdate`, but was accidentally omitted from `VendorUpdate`. Because `QBOInputModel` uses `extra="forbid"`, attempting to include `CompanyName` in a vendor update payload raised a `ValidationError`.

#### What was changed?

**Before:**
```python

    id: str = Field(alias="Id")
    sync_token: str = Field(alias="SyncToken")
    display_name: str | None = Field(default=None, alias="DisplayName")

    given_name: str | None = Field(default=None, alias="GivenName")
    family_name: str | None = Field(default=None, alias="FamilyName")
```

**After:**
```python
class VendorUpdate(QBOInputModel):
    id: str = Field(alias="Id")
    sync_token: str = Field(alias="SyncToken")
    display_name: str | None = Field(default=None, alias="DisplayName")
    active: bool | None = Field(default=None, alias="Active")
    given_name: str | None = Field(default=None, alias="GivenName")
    family_name: str | None = Field(default=None, alias="FamilyName")
    company_name: str | None = Field(default=None, alias="CompanyName")
```

#### How does the code work?

1. The new `company_name` field is an optional string, defaulting to `None`.
2. It uses the QBO API alias `CompanyName` for serialization.
3. When set, it serializes into the API payload; when `None`, it is excluded by `exclude_none=True`.

#### Why was this change needed?

Users could not update a vendor's company name through the typed API. This was an accidental omission from the bulk entity creation in the initial commit, where `company_name` was correctly included in `Vendor`, `VendorCreate`, and `CustomerUpdate` but missed on `VendorUpdate`.

### Bug 10: TimeActivityCreate missing customer_ref field
