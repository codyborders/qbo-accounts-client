### Bug 7: CLI `update` command crashes with AttributeError

**Detail Bug ID**: `bug_66517e8b-92a5-498b-aead-95c75f53b59b`
**File**: `qbo_accounts/cli.py`
**Security vulnerability**: No
**Impact**: High | **Severity**: Major | **Priority**: P0
**Status**: Fixed

#### What was the problem?

The CLI `update` command passed a raw Python dictionary directly to `resource.update(data)`. But the `update()` method on every resource class calls `data.model_dump(by_alias=True, exclude_none=True)` -- a method that only exists on Pydantic models, not on plain dictionaries. This meant that any user running `qbo update invoices '{"Id": "1", ...}'` from the terminal would get an `AttributeError: 'dict' object has no attribute 'model_dump'` crash.

#### What was changed?

**Before** (`qbo_accounts/cli.py`, lines 198-209):
```python
@main.command()
@click.argument("entity")
@click.argument("json_data")
def update(entity: str, json_data: str) -> None:
    """Update an entity from JSON data."""
    with _make_client() as client:
        resource = _get_resource(client, entity)
        _require_capability(resource, entity, "update")
        data = _parse_json(json_data)
        result = resource.update(data)
        _output(result)
```

**After**:
```python
@main.command()
@click.argument("entity")
@click.argument("json_data")
def update(entity: str, json_data: str) -> None:
    """Update an entity from JSON data."""
    with _make_client() as client:
        resource = _get_resource(client, entity)
        _require_capability(resource, entity, "update")
        data = _parse_json(json_data)
        if isinstance(resource, BaseResource):
            model = resource._update_cls.model_validate(data)
            result = resource.update(model)
        else:
            result = resource.update(data)
        _output(result)
```

A supporting change was also made in `qbo_accounts/resources/base.py` to add the `_update_cls` property:

```python
@property
def _update_cls(self) -> type[TUpdate]:
    """Resolve the concrete update-model type from generic args at runtime."""
    return type(self)._resolve_generic_arg(2, "_cached_update_cls")
```

#### How does the code work?

1. The `update` command receives a JSON string from the user and parses it into a Python dictionary via `_parse_json()`.
2. It checks whether the resource is a `BaseResource` instance. `BaseResource` is the generic base class that all standard QBO resources inherit from.
3. If it is, it calls `resource._update_cls` to get the correct Pydantic model class (e.g., `InvoiceUpdate`). The `_update_cls` property uses Python's generic type system to look up the third type argument from the class definition (e.g., `InvoicesResource(TransactionResource[Invoice, InvoiceCreate, InvoiceUpdate])` -- the third arg is `InvoiceUpdate`).
4. It then calls `.model_validate(data)` to convert the raw dictionary into a validated Pydantic model instance.
5. Finally, it passes the Pydantic model to `resource.update(model)`, which can safely call `.model_dump()` on it.

#### Why was this change needed?

Without this fix, the CLI's `update` command was completely broken for all entity types. The `create` command already had this pattern (converting dict to Pydantic model), but `update` was missing it. This was a P0 because it blocked all CLI update operations.

### Bug 1: Rate limiter ignores X-RateLimit-Reset when Retry-After is malformed

**Detail Bug ID**: `bug_37c6f12f-3edd-4167-9725-f6b88582c3d7`
**File**: `qbo_accounts/utils.py`
**Security vulnerability**: No
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

When the QBO API returned a rate-limit response with a malformed `Retry-After` header (not a valid number or HTTP date), the rate limiter was supposed to fall back to the `X-RateLimit-Reset` header. Instead, it skipped straight to the default 5-second delay. This happened because the old code had an early `return` statement that executed even when parsing failed, preventing the code from ever checking `X-RateLimit-Reset`.

#### What was changed?

**Before** (`qbo_accounts/utils.py`):
```python
def wait_if_needed(self, headers: dict[str, str]) -> None:
    retry_after = headers.get("Retry-After") or headers.get("retry-after")
    if retry_after:
        try:
            time.sleep(min(float(retry_after), MAX_RETRY_AFTER))
            return
        except (ValueError, TypeError):
            pass
        try:
            retry_dt = parsedate_to_datetime(retry_after)
            wait = max(0.0, retry_dt.timestamp() - time.time())
            time.sleep(min(wait, MAX_RETRY_AFTER))
        except (ValueError, TypeError):
            pass
        return  # <-- BUG: this return runs even when both parses fail

    reset = headers.get("X-RateLimit-Reset") or headers.get("x-ratelimit-reset")
    # ... never reached when Retry-After is present but malformed
```

**After**:
```python
def wait_if_needed(self, headers: dict[str, str]) -> None:
    retry_after = headers.get("Retry-After") or headers.get("retry-after")
    if retry_after:
        try:
            time.sleep(min(float(retry_after), MAX_RETRY_AFTER))
            return
        except (ValueError, TypeError):
            try:
                retry_dt = parsedate_to_datetime(retry_after)
                wait = max(0.0, retry_dt.timestamp() - time.time())
                time.sleep(min(wait, MAX_RETRY_AFTER))
                return
            except (ValueError, TypeError):
                pass  # Fall through to X-RateLimit-Reset

    reset = headers.get("X-RateLimit-Reset") or headers.get("x-ratelimit-reset")
    if reset:
        try:
            wait = max(0.0, float(reset) - time.time())
            time.sleep(min(wait, MAX_RETRY_AFTER))
            return
        except (ValueError, TypeError):
            pass

    time.sleep(DEFAULT_RETRY_DELAY)
```

#### How does the code work?

1. First, the method checks for a `Retry-After` header (the standard HTTP header for rate limiting).
2. It tries to parse it as a number (seconds to wait). If that works, it sleeps and returns.
3. If that fails (ValueError), it tries to parse it as an HTTP date string. If that works, it calculates the wait time and returns.
4. If both parses fail, the `pass` statement lets execution fall through -- no `return` happens.
5. Next, it checks for `X-RateLimit-Reset` (a non-standard but common header containing a Unix timestamp). If present, it calculates how long to wait and sleeps.
6. If nothing else works, it falls back to a default 5-second delay.

#### Why was this change needed?

The old code's misplaced `return` meant that when a server sent a garbage `Retry-After` value alongside a valid `X-RateLimit-Reset`, the client would only wait 5 seconds instead of the correct 30+ seconds. This could cause the client to immediately hit another rate limit, wasting API calls and potentially getting the client temporarily blocked.

### Bug 4: PaymentCreate.line should be required, not optional

**Detail Bug ID**: `bug_3c1cb316-ba27-4c5c-abff-b98b108f7084`
**File**: `qbo_accounts/models/transactions.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The `PaymentCreate` model defined the `line` field (which specifies which invoices the payment applies to) as optional (`list[dict] | None = Field(default=None, alias="Line")`). According to the QBO API documentation, `Line` is a required field when creating a payment. Allowing it to be `None` meant that users could create a `PaymentCreate` object without specifying any line items, and the QBO API would reject it with a confusing error.

#### What was changed?

**Before** (`qbo_accounts/models/transactions.py`, line 199):
```python
class PaymentCreate(QBOBaseModel):
    customer_ref: ReferenceType = Field(alias="CustomerRef")
    total_amt: float = Field(alias="TotalAmt")
    txn_date: str | None = Field(default=None, alias="TxnDate")
    line: list[dict] | None = Field(default=None, alias="Line")
```

**After**:
```python
class PaymentCreate(QBOBaseModel):
    customer_ref: ReferenceType = Field(alias="CustomerRef")
    total_amt: float = Field(alias="TotalAmt")
    txn_date: str | None = Field(default=None, alias="TxnDate")
    line: list[dict] = Field(alias="Line")
```

#### How does the code work?

1. By removing `| None` and `default=None`, the `line` field becomes a required field in the Pydantic model.
2. If a user tries to create a `PaymentCreate` without providing `line`, Pydantic will raise a `ValidationError` immediately -- before the API call is even made.
3. This catches the error early with a clear message ("field required") instead of letting a malformed request hit the QBO API.

#### Why was this change needed?

Required fields in the QBO API should be required in our models too. This prevents confusing API errors and gives users immediate feedback when they forget a required field.

### Bug 5: PreferencesUpdate missing all preference fields

**Detail Bug ID**: `bug_778d3819-ca0c-4cb9-9d3b-d628784101c9`
**File**: `qbo_accounts/models/system.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The `PreferencesUpdate` model only had `id` and `sync_token` fields. It was missing all 8 actual preference fields (like `AccountingInfoPrefs`, `SalesFormsPrefs`, etc.). This meant that users could not update any preferences -- the model would serialize to just `{"Id": "...", "SyncToken": "..."}` with no actual preference data.

#### What was changed?

**Before** (`qbo_accounts/models/system.py`):
```python
class PreferencesUpdate(QBOBaseModel):
    id: str = Field(alias="Id")
    sync_token: str = Field(alias="SyncToken")
```

**After**:
```python
class PreferencesUpdate(QBOBaseModel):
    id: str = Field(alias="Id")
    sync_token: str = Field(alias="SyncToken")
    accounting_info_prefs: dict | None = Field(default=None, alias="AccountingInfoPrefs")
    email_messages_prefs: dict | None = Field(default=None, alias="EmailMessagesPrefs")
    product_and_services_prefs: dict | None = Field(default=None, alias="ProductAndServicesPrefs")
    sales_forms_prefs: dict | None = Field(default=None, alias="SalesFormsPrefs")
    vendor_and_purchases_prefs: dict | None = Field(default=None, alias="VendorAndPurchasesPrefs")
    time_tracking_prefs: dict | None = Field(default=None, alias="TimeTrackingPrefs")
    tax_prefs: dict | None = Field(default=None, alias="TaxPrefs")
    currency_prefs: dict | None = Field(default=None, alias="CurrencyPrefs")
```

#### How does the code work?

1. Each preference field is typed as `dict | None` with `default=None`. This means they are all optional -- you only include the preferences you want to change.
2. Each field has an `alias` matching the QBO API's CamelCase key name (e.g., `accounting_info_prefs` serializes to `AccountingInfoPrefs`).
3. When `model_dump(by_alias=True, exclude_none=True)` is called, only the preferences the user actually set will appear in the JSON payload sent to QBO.

#### Why was this change needed?

Without these fields, the `PreferencesUpdate` model was useless -- it could only send an empty update request. Now users can update any combination of preferences in a single API call.

### Bug 2: CompanyInfo read uses duplicate realm_id in URL

**Detail Bug ID**: `bug_78d6b14f-0494-4255-93e8-b73db9403085`
**File**: `qbo_accounts/resources/system.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

The `CompanyInfoResource.read()` method called `self._client._build_path("companyinfo", self._client.realm_id)`. The `_build_path` method already includes the realm_id in the base path (e.g., `/v3/company/{realm_id}/`), so passing it again produced a URL like `/v3/company/1234567890/companyinfo/1234567890` instead of the correct `/v3/company/1234567890/companyinfo`.

#### What was changed?

**Before** (`qbo_accounts/resources/system.py`, line 75):
```python
def read(self) -> CompanyInfo:
    path = self._client._build_path("companyinfo", self._client.realm_id)
    resp = self._client.request("GET", path)
    return CompanyInfo.model_validate(resp[self.ENTITY_KEY])
```

**After**:
```python
def read(self) -> CompanyInfo:
    path = self._client._build_path("companyinfo")
    resp = self._client.request("GET", path)
    return CompanyInfo.model_validate(resp[self.ENTITY_KEY])
```

#### How does the code work?

1. `_build_path("companyinfo")` constructs the URL path `/v3/company/{realm_id}/companyinfo`.
2. The realm_id is already embedded in the base path by the client, so we don't need to pass it again.
3. The `CompanyInfo` endpoint is a singleton -- there's only one company info per QBO account, so no entity ID is needed.

#### Why was this change needed?

The duplicate realm_id would cause the QBO API to return a 404 Not Found error, making it impossible to read company information. This is a QBO-specific URL structure issue.

### Bug 3: query() doesn't strip duplicate pagination clauses

**Detail Bug ID**: `bug_52c11bed-1059-438e-a6e6-5cd9a8a98e5e`
**File**: `qbo_accounts/resources/base.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

If a user included `STARTPOSITION` or `MAXRESULTS` in their `where` clause (either intentionally or by copy-pasting a query), the `query()` method would append them again, producing a SQL string with duplicate pagination clauses like `... STARTPOSITION 1 MAXRESULTS 50 STARTPOSITION 1 MAXRESULTS 100`. The QBO API's behavior with duplicate clauses is undefined and could return unexpected results.

#### What was changed?

**Before** (`qbo_accounts/resources/base.py`, lines 123-125):
```python
def query(self, where=None, order_by=None, start_position=1, max_results=100):
    """Run a single-page SQL-like query."""
    sql = self._build_query(where, order_by)
    sql += f" STARTPOSITION {start_position} MAXRESULTS {max_results}"
```

**After**:
```python
from ..pagination import _PAGINATION_CLAUSE_RE, auto_paginate_query

def query(self, where=None, order_by=None, start_position=1, max_results=100):
    """Run a single-page SQL-like query."""
    sql = self._build_query(where, order_by)
    sql = _PAGINATION_CLAUSE_RE.sub("", sql).strip()
    sql += f" STARTPOSITION {start_position} MAXRESULTS {max_results}"
```

#### How does the code work?

1. `_PAGINATION_CLAUSE_RE` is a regular expression (already defined in `pagination.py`) that matches patterns like `STARTPOSITION 123` and `MAXRESULTS 456` (case-insensitive).
2. `.sub("", sql)` replaces any matches with an empty string, effectively removing any existing pagination clauses from the query.
3. `.strip()` cleans up any trailing whitespace left behind.
4. Then the method appends the correct `STARTPOSITION` and `MAXRESULTS` values from the function parameters.

#### Why was this change needed?

This is a defensive fix that prevents undefined behavior from duplicate pagination clauses. The `auto_paginate_query` helper in `pagination.py` already had this stripping logic, but the single-page `query()` method did not.

### Bug 6: ItemUpdate includes immutable Type field

**Detail Bug ID**: `bug_6f94c634-52bc-4bba-b0c4-33a83fd9496d`
**File**: `qbo_accounts/models/namelist.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

The `ItemUpdate` model included a `type` field (aliased to `Type`). In the QBO API, the `Type` field on an Item is immutable -- once set during creation, it cannot be changed. If a user included `Type` in an update payload, the QBO API would reject the request with a 400 Bad Request error.

#### What was changed?

**Before** (`qbo_accounts/models/namelist.py`, lines 145-152):
```python
class ItemUpdate(QBOBaseModel):
    id: str = Field(alias="Id")
    sync_token: str = Field(alias="SyncToken")
    name: str | None = Field(default=None, alias="Name")
    type: str | None = Field(default=None, alias="Type")
    active: bool | None = Field(default=None, alias="Active")
    unit_price: float | None = Field(default=None, alias="UnitPrice")
    description: str | None = Field(default=None, alias="Description")
```

**After**:
```python
class ItemUpdate(QBOBaseModel):
    id: str = Field(alias="Id")
    sync_token: str = Field(alias="SyncToken")
    name: str | None = Field(default=None, alias="Name")
    active: bool | None = Field(default=None, alias="Active")
    unit_price: float | None = Field(default=None, alias="UnitPrice")
    description: str | None = Field(default=None, alias="Description")
```

#### How does the code work?

1. The `type` field line was simply removed from the `ItemUpdate` class.
2. Since the model uses `exclude_none=True` during serialization, removing the field ensures `Type` can never appear in update payloads.
3. The `Item` read model and `ItemCreate` model still have the `type` field -- it's only excluded from updates.

#### Why was this change needed?

Including an immutable field in an update model is misleading. Users might set it thinking they can change an item's type, only to get a confusing API error. Removing it from the model makes the API surface honest about what can be changed.
